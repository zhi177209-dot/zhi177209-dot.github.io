STM32F103C8T6 / UART PROTOCOL LAB

串口协议温湿度采集系统

这个小项目的重点不是单纯读取一次 DHT11，而是练习从上位机发送协议帧、STM32 接收拆包、 校验命令、执行业务、返回结果与错误码的完整链路。串口命令触发后，系统会读取温湿度数据， 通过 USART1 返回 ASCII 文本，并同步刷新 OLED 显示。

PROJECT OVERVIEW

项目简介

上位机通过串口发送协议命令，STM32 接收并解析命令，然后控制 DHT11 采集温湿度数据， 同时把结果通过串口返回，并显示在 OLED 屏幕上。项目把“串口通信”和“传感器采集”串成了一个可验证的小闭环。

系统说明

工程使用 STM32F103C8T6。USART1 负责上位机通信，波特率为 115200，PA9 为 TX， PA10 为 RX；OLED 使用 I2C1，对应 PB6 和 PB7；DHT11 数据线连接 PA5。为了让 DHT11 的单总线时序更稳定， 程序使用 TIM1 构造微秒级延时。

程序上电后先初始化 GPIO、I2C1、TIM1、USART1 和 OLED，然后开启串口空闲中断接收。 收到数据后，中断回调只负责把原始字节写入环形缓冲区；主循环再调用协议解析函数， 根据解析状态执行读取、清屏、返回缓存数据或错误处理。

01

串口接收

使用 HAL_UARTEx_ReceiveToIdle_IT() 实现空闲中断接收，避免主循环阻塞等待串口数据。

02

协议解析

command.c 维护环形缓冲区，并按帧头、长度、校验和拆出有效业务数据。

03

业务执行

main.c 使用 switch(command[0]) 一帧只执行一个命令，后续字节可作为参数扩展。

04

错误反馈

err.c 统一返回 EE 开头的错误帧，并通过日志文本辅助定位问题。

UART FRAME DESIGN

串口协议设计

协议采用“帧头 + 总长度 + 业务数据 + 累加校验”的结构。正常命令帧以 AA 开头， 错误返回帧以 EE 开头，便于串口助手中快速区分请求和异常反馈。

AA

正常命令帧

结构为 AA LEN DATA SUM。其中 LEN 是整帧长度， DATA 的第一个字节作为命令码，SUM 是前面所有字节的累加和。

01

读取温湿度

AA 04 01 AF 触发 DHT11 采集。成功后串口返回 Humidity 与 Temperature 文本，OLED 同步显示。

02

清空 OLED

AA 04 02 B0 执行 OLED 清屏，适合演示上位机通过协议控制本地显示模块。

03

返回缓存数据

AA 04 03 B1 不重新采集传感器，直接返回上一次保存的 DHT11 数据。

void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size)
{
    if (huart == &huart1) {
        Command_Write(readBuffer, Size);
        HAL_UARTEx_ReceiveToIdle_IT(&huart1, readBuffer, sizeof(readBuffer));
    }
}

while (1)
{
    uint8_t command[50];
    uint8_t commandLength = 0;
    CommandStatus_t status = Command_GetCommand(command, &commandLength);

    switch (status) {
    case CMD_OK:
        switch (command[0]) {
        case 0x01:
            Log_Print("[CMD] Read DHT11\r\n");
            if (DHT11_ReadData(&DHT11_Data)) {
                snprintf(message, sizeof(message),
                    "Humidity:%d.%d%% Temperature:%d.%dC\r\n",
                    DHT11_Data.humi_int, DHT11_Data.humi_deci,
                    DHT11_Data.temp_int, DHT11_Data.temp_deci);
                HAL_UART_Transmit(&huart1, (uint8_t *)message, len, 1000);
                OLED_Clear();
                // OLED 显示“湿度”和“温度”以及数值
            } else {
                Send_Error(ERR_SENSOR_FAIL);
            }
            break;

        case 0x02:
            OLED_Clear();
            break;

        case 0x03:
            // 直接返回上一次 DHT11 数据
            break;
        }
        break;
    }
}
Core/Src/command.c 
#define FRAME_HEAD1 0xAA
#define COMMAND_TIMEOUT_MS 1000

CommandStatus_t Command_GetCommand(uint8_t *command, uint8_t *commandLength)
{
    static uint32_t frameStartTick = 0;
    uint8_t bufferLength = Command_GetLength();

    if (bufferLength == 0) {
        frameStartTick = 0;
        return CMD_NONE;
    }

    if (Command_Read(readIndex) != FRAME_HEAD1) {
        Command_AddReadIndex(1);
        return CMD_ERR_HEAD;
    }

    uint8_t length = Command_Read(readIndex + 1);
    if (length < COMMAND_MIN_LENGTH || length > PACK_BUF_MAX) {
        Command_AddReadIndex(1);
        return CMD_ERR_LENGTH;
    }

    if (bufferLength < length) {
        if (HAL_GetTick() - frameStartTick > COMMAND_TIMEOUT_MS) {
            Command_AddReadIndex(1);
            return CMD_ERR_TIMEOUT;
        }
        return CMD_NONE;
    }

    uint8_t sum = 0;
    for (uint8_t i = 0; i < length - 1; i++) {
        sum += Command_Read(readIndex + i);
    }

    if (sum != Command_Read(readIndex + length - 1)) {
        Command_AddReadIndex(length);
        return CMD_ERR_CHECKSUM;
    }
}
Core/Src/err.c 
#define ERR_TIMEOUT         0x01
#define ERR_CHECKSUM        0x02
#define ERR_INVALID_LENGTH  0x04
#define ERR_UNKNOWN_CMD     0x05
#define ERR_SENSOR_FAIL     0x06

void Send_Error(uint8_t err)
{
    uint8_t frame[4];

    frame[0] = 0xEE;
    frame[1] = 0x04;
    frame[2] = err;
    frame[3] = frame[0] + frame[1] + frame[2];

    HAL_UART_Transmit(&huart1, frame, 4, 1000);
}

void Log_Print(char *msg)
{
    uint16_t len = strlen(msg);
    HAL_UART_Transmit(&huart1, (uint8_t *)msg, len, 1000);
}

DEBUG NOTES

问题与解决方式

这部分整理自你的 program.docx。它记录了这个串口协议项目从“能跑” 到“能识别异常、能定位问题”的调试过程。

01

接收数据被重新封包

早期 Command_Write() 会把串口收到的数据再次调用封包函数，导致解析器校验的是自己重新封好的帧， 不是上位机发来的原始帧。后来改为只把原始字节写入环形缓冲区。

02

执行 02 时 01 又被执行

原因是主循环遍历了整个 50 字节数组，上一帧残留的 0x01 又被扫描到。 最终改成 switch(command[0])，保证一帧只执行一个命令。

03

串口中文乱码

数字正常但中文乱码，说明波特率没问题，主要是编码不统一。当前输出改成英文 ASCII： Humidity:%d.%d%% Temperature:%d.%dC，更适合串口助手稳定显示。

04

DHT11 采集失败

通过检查 PA5、VCC/GND、数据线上拉和上电稳定时间定位问题。程序中加入上电稳定延时， 并用 TIM1 保证微秒级时序后，复位后的读取更稳定。

05

校验错误后非法帧头刷屏

校验失败后如果只丢 1 字节，坏帧剩余字节会继续触发非法帧头。修正为校验失败时丢弃整包： Command_AddReadIndex(length)。

06

长度错误与超时错误区分

AA 03 01 AF 属于长度字段非法，因为小于最小帧长 4； AA 06 01 AF 属于长度合法但数据没收够，所以返回超时错误。

CONCEPT NOTES

知识点补充

这个小项目最值得保留的学习点，是把串口协议从“直接收字节”推进到了“帧结构、状态码和错误恢复”。

协议解析的基本顺序

  • 先找帧头：不是 AA 就丢 1 字节继续同步。
  • 再读长度：长度小于最小帧长或超过缓存上限，立即报长度错误。
  • 等待完整帧：长度合法但数据没收够，超过 1 秒后报超时。
  • 最后校验和：校验失败丢弃整包，避免剩余字节继续污染解析器。
  • 业务分发：解析成功后只把业务数据交给 switch(command[0])。
01

中断接收

串口中断负责接住原始字节，不在中断里做复杂业务，主循环再解析和执行。

02

错误帧

统一错误返回能让上位机快速判断问题类型，比只打印文字日志更适合协议化通信。
