STM32F103C8T6 / MINI LAB 02

小型温湿度采集与显示

这个小项目围绕 DHT11、OLED 和 USART1 展开， 用 STM32F103C8T6 完成温湿度采集、屏幕显示、串口输出以及按分钟周期记录。 它不是单一外设的孤立练习，而是一次把传感器时序、显示驱动、串口调试和定时任务串起来的完整小闭环。

PROJECT OVERVIEW

项目简介

这个项目的核心目标，是把温湿度数据从“读到”扩展到“看见”和“记录下来”。 DHT11 负责提供环境数据，OLED 负责本地显示，USART1 负责把同一份结果发到串口助手， TIM2 则把采样动作稳定地控制在 1 分钟一次。

系统说明

工程由 CubeMX 生成基础初始化代码，主程序先完成 GPIO、TIM1、TIM2、USART1 和 I2C1 的初始化， 随后启动 OLED、等待 DHT11 上电稳定，再由 TIM2 提供周期计数。程序上电后会先执行一次采样， 之后每累计到 1 分钟重新采集一次，并把同一份温湿度结果同时写到 OLED 和串口。

从结果上看，这不是一张静态展示图，而是一个持续运行的温湿度采集小系统。 我提供的 JSON 记录显示，系统在 2026-07-13 21:33:40 到 22:05:37 之间连续记录了 33 条房间数据， 这也让页面里展示的实物图和串口图具备了更强的真实性。

01

DHT11 采集

PA5 复用为单总线数据脚，主机先发送起始信号，再等待 DHT11 应答并读取 40bit 数据，最后通过校验和确认结果有效。

02

OLED 显示

OLED 通过 I2C1 与 MCU 通信，显示内容采用 16x16 汉字字模和 8x16 数字字模组合，直接在屏幕上渲染“湿度”“温度”和数值。

03

UART 输出

USART1 以 115200 8N1 输出 ASCII 文本，串口助手可实时看到每轮采样结果，方便联调和留存原始记录。

04

定时记录

TIM2 溢出中断只做计数，主循环按计数结果决定何时再次采样，避免把复杂业务逻辑塞进中断回调。

IMPLEMENTATION NOTES

关键实现思路

这一部分专门解释几个容易“看代码能跑，但不容易一下看懂”的实现细节。

01

DHT11 为什么切换输入/输出模式

DHT11 使用单总线协议，同一根数据线既要由主机发起启动信号，也要由传感器返回应答和数据。 因此 MCU 端必须先把引脚配置为输出，主动拉低总线发送起始信号；随后再切换为输入，释放总线，等待 DHT11 响应并输出 40 位数据。

02

为什么用 TIM1 做微秒延时

DHT11 不仅要求启动阶段保持低电平至少 18ms，还要求在释放总线后进行 20~40us 级等待， 并在大约 40us 的采样点判断当前位是 0 还是 1。为了让这些关键时序更稳定，TIM1 被配置为 1MHz 计数基准， 从而给 bsp_delay_us() 提供可靠的微秒级延时。

03

为什么读取阶段临时关中断

DHT11 单次读取 40bit 时，每一位的判定窗口只有几十微秒。如果刚好被 TIM2 之类的中断打断， 采样点就可能偏移，最终导致校验失败或读数异常。所以在关键读取时序阶段临时屏蔽中断，读完后再恢复， 能更稳地保证单总线时序完整。

04

为什么 OLED 采用字模显示

SSD1306 这类 OLED 本质上是点阵屏，不像串口终端那样天然支持中文字符串显示。 这次又需要显示“湿度”“温度”等汉字，所以更合适的方式是提前准备汉字点阵和数字点阵， 再按页地址把字节写入显存。这样既能稳定显示中文，也能精确控制排版和小数格式。

Core/Src/bsp_dht11.c 
uint8_t DHT11_ReadData(DHT11_Data_TypeDef *DHT11_Data)
{
    uint32_t primask;

    // 主机拉低总线，发送起始信号
    DHT11_Mode_OUT_PP();
    HAL_GPIO_WritePin(BSP_DHT11_PORT, BSP_DHT11_PIN, GPIO_PIN_RESET);
    HAL_Delay(18);

    primask = __get_PRIMASK();
    __disable_irq();

    // 释放总线，切到输入等待应答
    HAL_GPIO_WritePin(BSP_DHT11_PORT, BSP_DHT11_PIN, GPIO_PIN_SET);
    DHT11_Mode_IN_NP();
    bsp_delay_us(30);

    if (!DHT11_WaitForPinState(GPIO_PIN_RESET, 100)) {
        if (!primask) { __enable_irq(); }
        return 0;
    }
    if (!DHT11_WaitForPinState(GPIO_PIN_SET, 100)) {
        if (!primask) { __enable_irq(); }
        return 0;
    }

    DHT11_Data->humi_int = DHT11_ReadByte();
    DHT11_Data->humi_deci = DHT11_ReadByte();
    DHT11_Data->temp_int = DHT11_ReadByte();
    DHT11_Data->temp_deci = DHT11_ReadByte();
    DHT11_Data->check_sum = DHT11_ReadByte();

    DHT11_Mode_OUT_PP();
    HAL_GPIO_WritePin(BSP_DHT11_PORT, BSP_DHT11_PIN, GPIO_PIN_SET);
    if (!primask) { __enable_irq(); }

    return (DHT11_Data->check_sum ==
        (DHT11_Data->humi_int + DHT11_Data->humi_deci +
         DHT11_Data->temp_int + DHT11_Data->temp_deci)) ? 1U : 0U;
}
Core/Src/oled.c 
uint8_t OLED_Init(void)
{
    static const uint16_t probe_addrs[] = {OLED_DEV_ADDR_GND, OLED_DEV_ADDR_VCC};

    for (uint8_t i = 0; i < 2; i++) {
        if (HAL_I2C_IsDeviceReady(&hi2c1, probe_addrs[i], 2, 50) == HAL_OK) {
            oled_i2c_addr = probe_addrs[i];
            oled_ready = 1U;
            break;
        }
    }

    if (oled_ready == 0U) {
        return 0U;
    }

    OLED_WriteCmd(0xAE);
    OLED_WriteCmd(0xD5);
    OLED_WriteCmd(0xA8);
    OLED_WriteCmd(0xAF);
    OLED_Clear();
    return 1U;
}

void OLED_ShowZh16x16ByIndex(uint8_t x, uint8_t page, uint8_t index)
{
    if (oled_ready == 0U || index >= OLED_ZH16X16_COUNT) {
        return;
    }

    OLED_SetPos(x, page);
    for (uint8_t col = 0U; col < 16U; col++) {
        OLED_WriteData(oled_zh16x16[index][4U + col]);
    }

    OLED_SetPos(x, (uint8_t)(page + 1U));
    for (uint8_t col = 0U; col < 16U; col++) {
        OLED_WriteData(oled_zh16x16[index][20U + col]);
    }
}

DEBUG NOTES

问题与解决方式

这里保留了这次从“能跑”到“稳定运行”过程中最关键的几个排障点，内容整理自你的项目问题文档。

01

TIM1 微秒延时不准

DHT11 依赖微秒级判位时序，如果定时器不是按照 1us 计数，读取结果就容易出错。 最终把 TIM1 配成 1MHz 计数基准，也就是 72MHz 主频下分频设为 72-1，让 bsp_delay_us() 具备稳定性。

02

DHT11 启动与读取时序不稳

起始信号、释放总线、等待低高电平应答、读取每一位的采样点，顺序和时长都必须符合协议。 后来按协议重新梳理流程，并给等待逻辑补上超时保护，避免因时序异常直接卡死。

03

TIM2 中断打断采样

DHT11 单次读取 40bit 时，每一位只有几十微秒的判定窗口，中断插入会打乱采样点。 现在的做法是在关键读取阶段临时关中断，读完再恢复，避免 TIM2 周期计数打断单总线时序。

04

第一次采集失败

一开始 PA5 上电后的空闲态处理不对，第一次握手容易失败。后续把端口初始输出调整为高电平， 直接保证总线空闲态正确，不再把“第一次失败”当成预热步骤去规避问题。

05

OLED 数值显示错位和错值

之前既出现过小数传参错误，也出现过温度数字页地址写错、数字字模与页写法不匹配的问题。 后来统一改成“整数乘 10 加小数位”的传值方式，并重写 8x16 数字字模和显示函数，显示结果才稳定下来。

CONCEPT NOTES

知识点补充

这次项目之外，还顺手把几类常见 STM32 外设接口的使用场景和区别做了一次集中梳理。

01

UART

适合调试打印、模块通信和上位机收发，优点是使用简单、定位问题直观，缺点是点对点通信能力有限。

02

I2C

适合挂接 OLED、EEPROM、环境传感器等低速外设，只用两根线就能接多个设备，但速率和时序容错一般。

03

SPI

适合高速全双工通信，比如高速屏、Flash 或高速转换器，优点是速度高，代价是引脚占用更多、片选管理更复杂。

04

ADC

用于采集模拟量，比如光敏电阻、电位器、电池电压等，把连续电压信号转换成 MCU 可处理的数字量。

05

PWM

适合输出可调占空比波形，常见于电机调速、舵机控制、LED 调光和蜂鸣器发声等场景。

06

DMA

适合大批量数据搬运，常与 UART、SPI、ADC 配合使用，能减少 CPU 轮询负担，提高实时性和吞吐量。
